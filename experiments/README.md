# Experiments

Each subdirectory is a self-contained experiment that can be applied with
`kubectl apply -k <dir>`.

## Use cases


- [`uc1-rapid-disaster-response/`](./uc1-rapid-disaster-response/) —
  Single producer; **time-to-first-delivery** as a function of file size
  `S`, priority `P`, constellation size and (EDFS-only) replication
  factor `RF`. EDFS vs TUS.
- [`uc2-continuous-los-relay/`](./uc2-continuous-los-relay/) — Every sat
  produces one file under a **continuous random fault stream** (no
  `Destroy`). Headline KPI: delivery success rate ≥ 95%. EDFS vs TUS.
- [`uc3-priority-aware-routing/`](./uc3-priority-aware-routing/) — Same
  scenario at `priority ∈ {low, default, high}` on a deterministic
  orbital schedule. Does EDFS actually privilege high-priority files?
  EDFS only.
- [`uc4-sat-failure-pole/`](./uc4-sat-failure-pole/) — Producer takes a
  polar shot and is killed by a `Destroy` event **before** its first LOS
  with any GS. Does EDFS still deliver via ISL? EDFS vs TUS (TUS is a
  negative-control baseline).
- [`uc5-general-failure/`](./uc5-general-failure/) — 20% of sats each
  produce 5 files under a continuous non-`Destroy` fault stream.
  Long-running stress test for eventual consistency. EDFS only.

## How the UC run drivers work (`run.sh` + `tiers.yaml`)

Every `uc<N>-…/` directory follows the same layout, generated by the
`yass-uc-experiment-implement` skill. The five drivers share one skeleton
and differ only in `make_behaviours` (who produces, which faults) and in
the sweep dimensions declared in `tiers.yaml`.

### Files in a UC directory

- **`tiers.yaml`** — the sweep matrix, grouped into `tier_1`, `tier_2`,
  `tier_3`. Each entry is one run, e.g.
  `{ engine, sat_count, file_size, priority, rf }`. Tier 1 is the headline
  curve; tiers 2/3 are parameter-sensitivity and deep-dive subsets.
- **`_layouts/n<NN>.yaml`** — a ready-made cluster-scoped `Layout`
  (topology: which satellites + ground stations, their TLEs/positions),
  one per `sat_count`. Applied once and shared across runs. Regenerate
  with `../_common_/regenerate-uc-layouts.py`.
- **`_template/{00_namespace,02_experiment_definition,03_experiment}.yaml.tmpl`**
  — manifest templates with `${VAR}` placeholders filled by `envsubst`.
- **`_runs/`** — driver output (git-ignored); see "Outputs" below.
- **`run.sh`** — the driver itself.

### Invocation

```shell
./run.sh --tier <1|2|3|all> [--kubeconfig <path>] [--dry-run]
```

- `--tier` — which group from `tiers.yaml` to run (`all` = 1+2+3).
- `--kubeconfig` — wins over `$KUBECONFIG`; falls back to `~/.kube/config`.
  The script exits 4 if the file does not exist (so even a dry-run needs a
  real path).
- `--dry-run` — render YAML to `_runs/<run_id>/` only; **apply nothing**
  (skips even the Layout apply). Without it, the driver runs the
  experiment on the cluster the kubeconfig points at.

### What one run does

The driver **does run the experiment on the cluster** (unless
`--dry-run`). It applies the `_layouts/` once, then for every `tiers.yaml`
entry:

1. Build the `run_id` / namespace (e.g. `uc2-edfs-pdefault-n08-rf3`,
   `uc2-tus-n08`).
2. Pick the matching `_layouts/n<NN>.yaml`.
3. `make_behaviours` — walk the layout and emit one `Behaviour` per
   fsNode: a `yass-agent-periodic` producer or a `yass-agent-receive-only`
   relay/GS, plus a per-fsNode `hardwareEvents` fault schedule whose `seed`
   is `fnv32(run_id + fsNode)` so faults are reproducible across re-runs.
4. Select the engine containers — one TUS container, or the three EDFS
   containers (`edfs-engine` + `-node` + `-proxy`) with cluster keys and
   bootstrap peer.
5. Render the templates with `envsubst` into `_runs/<run_id>/`.
6. `kubectl apply` the namespace + ExperimentDefinition + Experiment.
7. Poll `Experiment.status.experimentState` until it reaches a terminal
   state (`Success` / `Failure` / `TimedOut` / `Errored`).
8. Bundle artefacts with `tools/yass-export/yass-export.sh`.
9. `kubectl delete ns` and pause briefly before the next run.

### Outputs (in `_runs/`)

The driver computes no KPI and draws no charts — its product is raw data
on disk:

- `driver.log` — one line per run (`rendered → … / applied / terminal
  state: X / export ok|failed`).
- `<run_id>/` — the rendered, re-runnable manifests
  (`00_namespace.yaml`, `02_experiment_definition.yaml`,
  `03_experiment.yaml`) plus `run.log` and a `kustomization.yaml`, so a
  single variant can be launched directly with
  `kubectl apply -k _runs/<run_id>/`.
- `kustomization.yaml` (in `_runs/`) — a global overlay aggregating every
  rendered variant; `kubectl apply -k _runs/` launches them all at once
  (independently of the driver's sequential apply/wait/delete cycle).
- `<exp>-<run_id>.tar.gz` — the `yass-export` bundle: `events.ods` +
  `events-csv/` (Loki events), `metrics-csv/*.csv` (Prometheus snapshots:
  delivery time, RAM, CPU, network RX/TX), and `experiment.yaml` (the live
  CR). This bundle is the per-run deliverable; EDFS-vs-TUS comparison and
  plots are produced separately from these CSVs.

`yass-export` needs the `events-exporter` and `prom-snapshot` binaries on
`$PATH` (or via `EVENTS_EXPORTER_BIN` / `PROM_SNAPSHOT_BIN`) and a
reachable Prometheus/Loki; if they are missing the bundle step fails and
the run leaves only its rendered manifests and `driver.log` entry.

## Supporting / older experiments

- [`spain-shot/`](./spain-shot/) — single-orbit scenario over Madrid:
  one ONEWEB satellite takes a 2 G photo, seven ESTRACK ground stations
  receive. Has `base/`, `tus/`, `edfs/` overlays plus a `hwfaults/`
  variant that exercises every hardware-fault type in seconds rather
  than minutes (used as the canonical smoke-test for the
  hardware-events injector).
- [`forever/`](./forever/) — long-running constellation (3 satellites +
  7 ESTRACK ground stations) used to exercise sustained data flow over
  the simulated network. Has `edfs/` and `tus/` variants on top of a
  shared `base/` layout.
- [`scaling/`](./scaling/) — delivery time to a dedicated ground station
  as a function of constellation size (`n ∈ {1, 2, 3, 5, 8}`), TUS vs
  EDFS. Per-n overlays + sequential runner.
- [`big-scale/`](./big-scale/) — 60-satellite constellation, used to
  exercise the simulator and observability at non-trivial cardinality.
- [`networking-demo/`](./networking-demo/) — minimal connectivity demo
  (3 satellites + 1 ground station, `fs_engine_udp_ping` engine).
  Verifies that `world-controller` actually shapes traffic with tc.
- [`_common_/`](./_common_/) — shared `HardwareDefinition` set; not an
  experiment, included as a base.

## Adding a new experiment

Copy the closest existing directory (start from one of the `spain-shot/`
variants if you want hardware events; from `forever/` for a quiet
long-running scenario), edit the `Layout` / `ExperimentDefinition` /
`Experiment`, and update its `kustomization.yaml`. For a UC-style
parameter sweep, follow the `uc<N>-…` README's "TUS parameter coverage"
convention so the run-id matrix stays consistent.

**Don't forget the namespace label.** The `Namespace` YAML must carry
`labels.yass-namespace: "true"` — otherwise the operator's
namespace controller ignores it and no per-experiment infrastructure
(messaging broker, executor, mqtt2prom, …) is created. See the main
[README's "Namespace requirement"](../README.md#namespace-requirement)
section for the full reasoning.
